# Day 1：最小闭环版本

对应文件：`request_ex.py`。

## 1. 这个版本在做什么

Day 1 的任务只有一个：

**证明“API 数据可以被拉下来，再写成文件”。**

代码结构是：

* `fetch_users(url: str) -> list`
* `extract_rows(users: list) -> tuple[list, int]`
* `write_csv(path: str, rows: list) -> None`
* `main()`

也就是：

1. 发请求
2. 拿 JSON
3. 提取几个字段
4. 写入 CSV

这是一个非常典型的“**先把水管接通**”版本。

## 2. Day 1 改了什么

它已经做了几件对的事：

### 第一，已经有函数拆分

虽然还是很初级，但至少不是所有逻辑都堆在 `main()` 里。
它已经把“抓数据”“提字段”“写 CSV”分开了。

### 第二，已经有最基本的坏数据处理

在 `extract_rows()` 里，遇到缺字段的用户记录会 `skipped += 1`，并打印 warning。

### 第三，已经有最基本的异常分类

`main()` 里区分了：

* `requests.RequestException`
* `ValueError, OSError`
* `Exception`

这说明它不是“能跑就行”的最低级脚本，而是已经有一点“真实执行环境”的意识。

## 3. Day 1 的局限在哪里

问题不在于它“错”，而在于它还只是**脚本思维**，不是**工具思维**。

### 局限一：返回值不统一

* `fetch_users()` 返回 `list`
* `extract_rows()` 返回 `tuple[list, int]`
* `write_csv()` 返回 `None`

这意味着每个函数都要靠你自己记忆“它到底回什么”。
脚本小的时候没问题，脚本一旦变长，就会越来越乱。

### 局限二：成功与失败的信息分散

失败时靠 `raise` 或 `except`。
成功时靠 `print`。
跳过了几条记录，靠 `tuple` 里的第二个元素。
这导致“状态信息”没有一个统一容器。

### 局限三：字段还比较少

只有：

* `id`
* `name`
* `email`
* `company_name`

这对于练概念够了，但离客户世界常见的“稍微完整一点的表格”还有一点距离。

## 4. Day 1 我希望你掌握什么

Day 1 不是让你背库，而是让你掌握这 3 个概念：

### 概念一：API 返回的是结构化数据

不是一坨文本，而是 JSON，可以变成 Python 对象。

### 概念二：导出文件前，通常要做“平铺”

因为 API 里的数据结构常常是嵌套的，比如：

* `user["company"]["name"]`

而表格需要的是平的列。
所以“**嵌套 JSON → 平铺 row dict**”是第一关键动作。

### 概念三：一个小自动化工具至少有三层

* 输入：API
* 中间层：提取/转换
* 输出：CSV

你 Day 1 学的是这个骨架。

# Day 2：从脚本升级到“会说自己成功还是失败”

对应文件：`request_ex-D2.py`。

## 1. Day 2 为什么要改

Day 1 最大的问题不是功能不够，而是：

**程序虽然能跑，但它不会稳定地告诉你“自己现在处于什么状态”。**

这在真实交付里很要命。
因为客户不是只关心“能不能跑出来”，还关心：

* 是哪一步失败的
* 为什么失败
* 成功时到底输出了什么
* 写了多少行
* 有没有跳过坏数据

所以 Day 2 的核心思路是：

**给每一步都套一个统一的“结果壳子”。**

## 2. Day 2 改了什么

### 第一，引入最小 Result dict

`fetch_users()` 和 `save_users_to_csv()` 都开始返回这种结构：

* `ok`
* `stage`
* `reason`
* `evidence`

这一步极其关键。
因为从这一刻开始，你不再只是“拿到函数返回值”，而是在拿“**带状态的结果对象**”。

### 第二，字段更完整

Day 2 比 Day 1 新增了：

* `phone`
* `website`

这让输出更像真实业务表格。

### 第三，控制流更像小工具

`main()` 不再只是 try/except 一把梭，而是：

1. 调 `fetch_users()`
2. 看 `ok`
3. 再调 `save_users_to_csv()`
4. 再看 `ok`

也就是说，程序开始进入“**阶段式控制流**”。

## 3. Day 2 的进步本质是什么

这一天最大的进步，不是多了两个字段，也不是代码更长。

真正的进步是：

**你开始把“执行结果”对象化。**

这在后面会直接连接到：

* 更稳定的日志
* 更清晰的错误处理
* 更像产品的控制流
* 更容易扩展成多阶段流程

这对 Upwork 很重要，因为客户最怕的是：

* 代码 silent failure
* 跑错了但不知道哪一步错
* 输出不完整但没有证据链

Day 2 已经开始向“可交付”靠近。

## 4. Day 2 的问题在哪里

它不是终点，因为它有两个结构性问题。

### 问题一：`evidence` 被混用了

在 `fetch_users()` 成功时，`result["evidence"] = data`，也就是把整份原始 API 数据塞进了 `evidence`。
但在 `save_users_to_csv()` 里，`evidence` 又变成了统计信息：

* `rows_written`
* `skipped`
* `output_file`

这就造成一个问题：

**同一个字段 `evidence`，有时装原始数据，有时装摘要信息。**

这会导致语义不稳定。
也就是说，你以后读这个字段时，不知道里面到底是什么。

### 问题二：职责还是混的

`save_users_to_csv(users)` 一边提字段，一边写 CSV。
这说明“转换层”和“输出层”还没真正分开。

这会带来什么问题？

如果以后客户说：

* 不要 CSV，改成 Excel

你就会发现：
你不只是要改“文件输出”，还得碰“字段提取”逻辑。
这说明结构还不够干净。

## 5. Day 2 我希望你掌握什么

Day 2 你必须掌握两个思想：

### 思想一：函数不只是返回数据，还可以返回“状态 + 解释”

这比单纯 `return rows` 更接近真实工具。

### 思想二：状态容器一旦建立，语义必须稳定

一个字段最好只承担一种角色。
否则程序越大越混乱。

# Day 3：结构真正清楚的版本

对应文件：`request_ex-D3.py`。

## 1. Day 3 为什么要改

Day 2 让程序“会说话”了，但还没有真正“分层”。

所以 Day 3 的核心目标很明确：

**把“取数据”“转数据”“写文件”三件事彻底拆开。**

## 2. Day 3 改了什么

### 第一，`data` 和 `evidence` 分开

这是 Day 3 最重要的一刀。

Result 变成：

* `ok`
* `stage`
* `reason`
* `evidence`
* `data`

并且语义被明确了：

* `data`：真正要传给下一步的东西
* `evidence`：辅助说明信息，不负责传主数据

比如：

* `fetch_users()` 把原始 API 数据放到 `data`
* `evidence` 只放 `{"count": len(data)}`
* `extract_rows()` 把 rows 放到 `data`
* `evidence` 放 `rows_count`、`skipped`
* `save_rows_to_csv()` 不再产出下一步数据，所以 `data` 保持 `None`。

这是一次非常关键的语义收口。

### 第二，新增 `extract_rows(users)`

Day 3 不再用 `save_users_to_csv(users)` 同时提字段和写文件。
而是拆成：

* `fetch_users()`
* `extract_rows(users)`
* `save_rows_to_csv(rows)`

这意味着三层职责终于真正清楚了。

### 第三，配置开始集中

顶部开始有：

* `USERS_URL`
* `OUTPUT_CSV`
* `TIMEOUT`

这说明程序开始从“随手脚本”往“可调参数的小工具”走。

### 第四，输出细节更贴近 Win11/Excel

CSV 写入时用的是 `encoding="utf-8-sig"`。

这不是花哨点，而是很实用的 Windows/Excel 兼容意识。
它说明这个版本开始具备一点“客户环境意识”。

## 3. Day 3 的进步本质是什么

如果说 Day 2 是让程序“有状态”，
那 Day 3 就是让程序“**有数据流边界**”。

也就是说，你现在已经能清楚区分：

### 层 1：抓取层

负责从 API 拿原始数据。
不关心文件格式。

### 层 2：转换层

负责把嵌套 JSON 变成平铺 rows。
不关心最终写 CSV 还是 Excel。

### 层 3：输出层

负责把 rows 写入某个文件。
不关心 rows 是怎么来的。

这就是 Day 3 最有价值的地方。

## 4. Day 3 剩下的小问题

### 小问题一：`TIMEOUT = 20` 稍长

功能上没错，只是你当前练习阶段 10 秒通常更够用。

### 小问题二：成功时 `reason` 仍然带一点“状态说明”味道

比如：

* `Users fetched successfully.`
* `Rows extracted successfully.`

它不算错误，只是语义还不够严苛。
但这个阶段没必要为此继续重构。

## 5. Day 3 我希望你掌握什么

Day 3 你最该真正记住的是这条数据流：

**原始 API 数据 → 平铺 rows → 文件输出**

以及这条 Result 规则：

**主数据走 `data`，辅助说明走 `evidence`。**

这是你后面所有小工具的通用骨架。

# Day 4：输出层替换，主干不动

对应文件：`request_ex-D4.py`。

## 1. Day 4 为什么要改

到了 Day 3，其实结构已经够用了。
这时候再继续重构，收益很低。

所以 Day 4 的正确方向不是继续“结构洁癖”，而是验证一个更重要的能力：

**同样的数据流，能不能只替换输出层？**

也就是：

* 前两层不动
* 把 CSV 换成 Excel

这是更贴近 Upwork 的问题，因为客户常常不是改业务逻辑，而是改交付格式。

## 2. Day 4 改了什么

### 第一，输出文件从 CSV 变成 XLSX

顶部配置变成：

* `OUTPUT_XLSX = "users.xlsx"`

写文件方式从 `csv.DictWriter` 变成 `openpyxl.Workbook()`。

### 第二，输出函数替换为 `save_rows_to_excel(rows)`

它做的事情很明确：

1. 创建 workbook
2. 取 active worksheet
3. 设置 sheet 名为 `"users"`
4. 写表头
5. 按行 append
6. 保存文件

这说明你已经开始掌握最基本的 Excel 输出路径。

### 第三，前两层几乎不变

`fetch_users()` 和 `extract_rows(users)` 基本沿用了 Day 3 的结构和 Result 语义。

这恰恰是 Day 4 最成功的地方：

**它没有乱改前面的主干。**

## 3. Day 4 的进步本质是什么

Day 4 真正想让你掌握的，不是 `openpyxl` 的 API 本身。

而是这句话：

**输出格式是可替换层。**

也就是说：

* API 获取逻辑不需要改
* JSON 平铺逻辑不需要改
* 只要最后的输出函数换掉，就可以换交付形式

这在真实接单里特别有价值。
因为很多需求本质只是：

* “原来导出 CSV，改成 Excel”
* “原来写本地文件，改成上传”
* “原来写单个文件，改成多 sheet”

你如果主干设计清楚，这类需求就只是替换某一层，不会牵一发而动全身。 

## 4. Day 4 还存在什么小问题

### 小问题一：`headers` 还在函数内部

这不是 bug，只是说明字段配置还没完全收口。

### 小问题二：`except Exception as exc` 抓得比较宽

工程上偏粗，但当前阶段可接受。
因为 Day 4 的任务重点是“先把 Excel 输出能力建立起来”。

## 5. Day 4 我希望你掌握什么

你要掌握的不是“Excel 好复杂”，而是这 4 个最小动作：

1. `Workbook()`
2. `wb.active`
3. `ws.append(...)`
4. `wb.save(...)`

以及更重要的一点：

**当主干分层清楚时，替换输出目标是一件很轻的事。**

# 这四天我真正希望你掌握的内容

不是四份代码本身，而是下面这 6 个能力。

## 1. API 数据不是终点，只是原料

你不是为了“拿到 JSON”而拿到 JSON。
你是为了把它变成客户能用的结果。

## 2. 真实工具通常有三层

* 获取
* 转换
* 输出

一旦这三层混了，后面一改格式就很痛苦。

## 3. Result 对象是小工具的最小工程化入口

不用上来就 class、logging、框架。
一个稳定的 dict 结果结构，已经能解决很多问题。

## 4. `data` 和 `evidence` 必须分工

这是 Day 3 最重要的工程意识。
否则程序会越写越糊。

## 5. 输出层应该尽量可替换

CSV、Excel、JSON，本质上都只是“交付壳”。
你的核心价值是前面的数据流。

## 6. 你现在学的不是“高深后端”

你在学的是：

**客户能理解、能交付、能改需求的小型自动化工具能力。**

这正是 Sprint3 的目标。
